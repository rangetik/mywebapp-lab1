## 💻 2. Документація застосунку

**MyWebApp** — це RESTful сервіс на Node.js (Express), призначений для управління базою товарів (Inventory).

### Як налаштувати середовище розробки (локально):
Якщо ви хочете запустити застосунок без скриптів автоматизації:
1. Встановіть **Node.js (v20+)** та **PostgreSQL**.
2. Склонуйте репозиторій та встановіть залежності: `npm install`.
3. Створіть базу даних `mywebapp` та користувача `app`.
4. Запустіть міграції: `node src/db/migrate.js`.
5. Встановіть шлях до конфігу: `export CONFIG_PATH=./config/config.json` та запустіть: `node src/app.js`.

### Системні вимоги:
* **Node.js:** v20.x або новіше
* **npm:** v10.x або новіше
* **PostgreSQL:** v15 або новіше
* **Nginx:** v1.24 або новіше

### Конфігурація (`/etc/mywebapp/config.json`):
```json
{
  "database": {
    "host": "127.0.0.1",
    "port": 5432,
    "user": "app",
    "password": "12345678",
    "database": "mywebapp"
  },
  "server": {
    "port": 5200
  }
}
```

## 🔌 3. Документація API-ендпоінтів

Сервіс реалізує механізм **Content Negotiation**: формат відповіді залежить від заголовка `Accept`.

| Метод | Ендпоінт | Опис | Accept Header | Результат |
|-------|----------|------|---------------|-----------|
| GET | `/items` | Список усіх товарів | `application/json` | JSON-масив об'єктів |
| GET | `/items` | Список усіх товарів | `text/html` | HTML-таблиця |
| POST | `/items` | Створення товару | `application/json` | 201 Created |
| GET | `/items/:id` | Деталі товару по ID | `application/json` | Об'єкт товару (або 404) |

## 🧪 5. Інструкція з тестування

### 1. Перевірка Nginx та бізнес-логіки (Content Negotiation)

Перевірка виконується за допомогою `curl` з хост-машини або локально в терміналі:

**JSON формат:**
```bash
curl -i -H "Accept: application/json" http://localhost/items
```

**HTML формат (таблиця):**
```bash
curl -i -H "Accept: text/html" http://localhost/items
```

**Створення запису (POST):**
```bash
curl -X POST -H "Content-Type: application/json" -d '{"name":"Laptop", "quantity":5}' http://localhost/items
```

### 2. Перевірка прав доступу (Users & RBAC)

* **Student & Teacher:** Мають повний доступ до `sudo`. Для `teacher` встановлено примусову зміну пароля при першому вході (`chage -d 0`).
* **Operator:** Має обмежені права. Дозволено лише керування сервісом та Nginx без запиту пароля:

```bash
su - operator
sudo systemctl status mywebapp  # Дозволено
sudo apt update                 # Буде відмовлено
```

* **App User:** Системний користувач без оболонки входу (`/usr/sbin/nologin`).

### 3. Системні перевірки та звіти

**Gradebook:** Перевірка файлу з автоматично нарахованими балами:
```bash
sudo -u student cat /home/student/gradebook
```

**Nginx Status:** Перевірка, що Nginx активний та завантажив конфігурацію:
```bash
sudo nginx -t && sudo systemctl status nginx
```

**Власник процесу:** Перевірка, що застосунок запущено саме від імені користувача `app`:
```bash
ps aux | grep node
```


## 🐳 Лабораторна 2. Контейнеризація та оркестрація середовища

Тут описується процес автоматизації розгортання проєкту за допомогою **Docker** та **Docker Compose**. Уся інфраструктура застосунку (веб-сервер, бекенд та реляційна база даних) була ізольована в окремі контейнери, які працюють у виділеній приватній мережі.

### 🏗️ Компоненти архітектури системи

| Сервіс | Базовий образ | Зовнішній порт | Роль в інфраструктурі |
| :--- | :--- | :--- | :--- |
| **`nginx`** | `nginx:alpine` | `0.0.0.0:80` | Зворотний проксі (Reverse Proxy), захист внутрішніх ендпоінтів |
| **`web`** | `node:20-alpine` (локальна збірка) | *Прихований* (5200) | Бекенд застосунок на Express (бізнес-логіка) |
| **`db`** | `postgres:15-alpine` | *Прихований* (5432) | Персистентне сховище даних (PostgreSQL) |

---

### 🚀 Швидкий запуск (Quick Start)

Щоб підняти весь стек сервісів на локальній машині або віртуальному сервері, виконайте наступні команди:

```bash
# Клонування репозиторію (якщо проєкт розгортається з нуля)
git clone https://github.com/rangetik/mywebapp-lab1.git
cd mywebapp-lab1

# Запуск автоматичної збірки та підняття контейнерів у фоновому режимі
docker-compose up -d --build
```

Що відбувається під час першого запуску:
-Ініціалізується ізольована bridge-мережа проєкту mywebapp-lab1_kpi-custom-network.
-Створюється незалежний дисковий том pg_database_data для персистентного зберігання бази даних.
-Завантажуються мінімалістичні Alpine-образи для PostgreSQL та Nginx.
-Збирається локальний образ для Node.js додатка (копіюється код, встановлюються production-залежності).
-Контейнер web перед стартом сервера автоматично виконує внутрішній скрипт міграції (node src/db/migrate.js), створюючи таблицю inventory.
-Запускається веб-проксі Nginx, який починає приймати зовнішній трафік на 80-му порту.

🔬 Інструменти тестування API (Верифікація)

```bash
# 1. Перевірка доступності головної сторінки (відповідь від Express через Nginx)
curl http://localhost/

# 2. Перегляд усього списку елементів (запит зчитує дані з PostgreSQL)
curl http://localhost/items

# 3. Перевірка безпеки: доступ до сервісних роутів ззовні заблоковано (очікується 403 Forbidden)

```

🛠️ Особливості реалізації збірки
- Оптимізація Dockerfile: Для Node.js застосунку обрано ультра-легкий базовий образ node:20-alpine. Копіювання файлів package*.json та виконання команди npm install --only=production винесено вище копіювання самого коду додатка. Це дозволяє Docker кешувати шар із залежностями — при зміні логіки в src/ важкий етап завантаження пакетів з npm не повторюється.

- Внутрішній DNS: Оскільки контейнери знаходяться в одній приватній мережі, замість жорсткого прописування IP-адрес застосовано вбудований Docker DNS. Бекенд звертається до СКБД за системним ім'ям хоста "host": "db", а Nginx перенаправляє проксі-запити на "proxy_pass http://web:5200".
